# ADR-001 — Torch-only v1 rewrite replaces PyOpenCL path

## Status

Accepted

## Date

2026-04-18

## Context

The project is a greenfield rewrite of CellModeller. During bootstrap, there is pressure to either:

1. emulate legacy PyOpenCL structure for familiarity,
2. build a Torch-first implementation with minimal API,
3. introduce an abstract multi-backend layer early.

Current repository implementation already provides a deterministic toy engine on top of PyTorch and does not include a PyOpenCL runtime path.

## Decision

Adopt and maintain a **Torch-only v1** engine strategy.

- PyOpenCL is intentionally replaced, not wrapped.
- No compatibility layer for legacy OpenCL execution is included.
- No generalized multi-backend abstraction is introduced during bootstrap.

## Rationale

- Keeps early architecture and tests simple.
- Aligns with current implemented API and code.
- Supports deterministic CPU-first validation.
- Avoids carrying legacy engine baggage into new module boundaries.

## Current implemented scope this ADR governs

- Public API: `Simulation`, `SimulationConfig`, `SimulationState`
- Engine: deterministic Torch-based stepping (`positions += velocities * dt`)
- CPU-first correctness baseline
- Scaffolded (not implemented) domains for mechanics/fields/biology

## Consequences

### Positive

- Faster iteration on a truthful minimal core.
- Clear contributor expectations about backend direction.
- Lower maintenance burden in bootstrap phase.

### Negative / tradeoffs

- No immediate legacy-runtime parity path.
- Backend portability is deferred.
- CUDA support remains bounded by PyTorch behavior and project maturity.

## Alternatives considered

### A) Direct PyOpenCL-to-Torch port
Rejected: preserves old architecture constraints and slows modernization.

### B) Early multi-backend abstraction
Rejected: adds complexity before validated need.

### C) Torch-only v1 (chosen)
Chosen: matches current implementation and bootstrap goals.

## Revisit triggers

Revisit this ADR only if one of the following becomes concrete:

- hard requirement to execute legacy PyOpenCL model paths,
- clear need for a second backend validated by real users,
- Torch limitations that block required simulation behavior.
