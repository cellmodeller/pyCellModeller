# ADR-001 — Build pyCellModeller as a Torch-first rewrite, not a PyOpenCL port

## Status

Accepted

## Context

The project goal is to create a Python-native reimplementation of CellModeller that replaces the historical PyOpenCL/OpenCL execution approach.

There are at least three plausible directions:

1. Port the old engine concepts directly and swap low-level calls
2. Build a new Torch-first engine with modern package boundaries
3. Attempt a generalized multi-backend architecture from the start

The project needs to move quickly toward a maintainable, testable, modern codebase.

## Decision

Build `pyCellModeller` as a **greenfield Torch-first rewrite**.

Do not treat the old repository layout or PyOpenCL implementation patterns as the template.

## Options considered

### Option A — Direct PyOpenCL-to-Torch port
Pros:
- may preserve historical logic more literally
- may feel closer to old implementation details

Cons:
- imports legacy architectural baggage
- risks preserving poor package boundaries
- slows modernization
- makes testing and API design harder

### Option B — Greenfield Torch-first rewrite
Pros:
- clean architecture from day one
- easier to test
- easier to package
- clearer separation of concerns
- aligns naturally with current ML tooling

Cons:
- requires rethinking some historical implementations
- may delay exact legacy feature parity

### Option C — Generalized multi-backend framework from the start
Pros:
- future flexibility
- theoretical portability

Cons:
- too much upfront abstraction
- slows delivery
- unclear real need at bootstrap stage
- higher maintenance burden early

## Why this option is best now

The main modernization step is already large: replacing PyOpenCL with PyTorch. The fastest path to a credible foundation is a clean Torch-first architecture with CPU correctness and future CUDA support, rather than preserving historical implementation baggage or abstracting too early.

## Consequences

- The package will not mirror the historical code structure.
- PyTorch device semantics become central to engine design.
- CPU correctness is the reference path.
- Historical compatibility is secondary to clarity and maintainability.
- If multi-backend support is wanted later, it should be justified by real needs and captured in a future ADR.

## Risks and mitigations

### Risk
Loss of exact behavioral parity with historical implementations.

### Mitigation
Use representative legacy examples as regression targets rather than porting architecture blindly.

### Risk
Torch-specific choices may become too entrenched.

### Mitigation
Keep domain models and public API clean, even while using Torch as the engine.

## Revisit triggers

Revisit this decision if:
- exact legacy compatibility becomes a hard requirement
- Torch proves inadequate for key simulation kernels
- there is a concrete need for a second execution backend
